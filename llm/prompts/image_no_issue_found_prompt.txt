==================== 1) SPAM (STRICT, HIGH THRESHOLD) ====================
Use Spam ONLY when screenshots provide NO meaningful task-related evidence.

Spam means:
- The case never enters a valid execution context for the task.

Critical clarification:
- If the run stays on the WRONG surface for the entire run, then opening menus or clicking things inside that wrong surface is NOT task progress.
  Example: opening a menu on an MSN webpage card does not count as progress for a Windows Widgets dashboard task.

Definition (critical):
- "Meaningful task-related evidence" means evidence of REAL task progress, such as opening the required surface (widgets panel/settings/task manager/etc.)
  OR performing at least one of the standard actions that changes UI state (menus/panels/dialogs opened, toggles changed, required page reached).
  Merely showing a static baseline screen (e.g., Edge new tab/MSN feed/desktop) across multiple steps is NOT meaningful task evidence.
  A required entry point being VISIBLE but never actually opened/clicked (e.g., three-dots menu, "Personalize", a settings button) is ALSO NOT meaningful evidence.
  Seeing a generic baseline that could exist without doing anything (e.g., an Edge window sitting on New Tab/MSN, a desktop/taskbar/search box without any popup/menu) does NOT count as a real attempt.
  A real attempt must show at least ONE non-trivial checkpoint/state change tied to the standard (menu/popup/pane/banner opened, a specific URL loaded, a system surface opened).

Important examples (baseline is NOT meaningful evidence):
- If the standard requires a hover-expanded popup/titlebar (desktop/taskbar search), then merely showing the search box or an image/icon inside it is baseline-only evidence.
  If the hover popup/titlebar never appears in any screenshot, this is Spam.
- If the standard requires an Edge notification banner under the address bar (or Task Manager end-task), then merely showing Edge open on a start/new tab/MSN page is baseline-only evidence.
  If the banner/Task Manager never appears in any screenshot, this is Spam.

Non-negotiable clarification:
- A visible entry button (e.g., "Personalize") without the popup/page ever being opened is NOT progress.
  Do NOT treat this as Context=relevant or Progress=partial.
  If the run stays on the baseline scene and never opens the required popup/pane/page/system surface, this is Spam.

HARD Spam triggers:
- All screenshots are essentially the SAME (or only trivial loading/carousel changes)
  AND there is NO evidence of any task actions being performed.
- All screenshots stay in the SAME scene/surface (same app/page/module context)
  AND the only visible differences are carousel rotations (or no visible change at all)
  AND there is NO step-level UI interaction shown (no menus/panels/popups/dialogs/banners/system UI opened) => Spam.
- Screenshots remain on welcome / new tab / MSN feed / loading / blank / setup only
  with no task-specific navigation (e.g., never reaches the required settings/module/page).
- Required NON-browser surface is never shown at all (when the standard requires it)
  e.g., Windows Widgets panel, Task Manager, system Settings, desktop search bar, etc.
- Required dynamic checkpoint is never shown (even if the baseline surface is visible):
  e.g., standard requires a hover-expanded popup/titlebar or a banner/popup/pane, but screenshots only show the baseline surface with none of those dynamic UI elements ever appearing => Spam.

Additional Spam triggers (derived from common Spam->Incorrect failures):
- "Wrong surface for the whole run": the standard requires a NON-browser/system surface (e.g., Widgets via Win+W, Windows Settings, Task Manager, desktop search bar),
  but ALL screenshots are only a browser/MSN webpage (or only plain desktop) with no sign of entering that surface.
- "Loading-only evidence": screenshots are not identical but only show page loading/refreshing states without reaching the required UI/checkpoints.
  Treat this as Spam (no meaningful progress).
- "Baseline-only in browser": the standard requires opening a menu/popup/pane/notification (e.g., Edge 3-dots menu, History pane, Personalize popup, settings panel, import notification banner),
  but ALL screenshots only show the main browser page (often Edge new tab/MSN feed) with NO menu/popup/pane/banner ever visible.
  If there is no meaningful UI state change across steps, treat as Spam (no real attempt), even if the entry button is visible.
- "Entry point visible but never used": if a required entry button is visible (e.g., three-dots menu, "Personalize", "Settings") but it is NEVER clicked/opened in any step,
  and the run stays on baseline pages with no state change, treat as Spam rather than Incorrect.

Decisive Spam rule (to prevent mislabeling as Incorrect):
- If ALL screenshots stay on a clearly irrelevant surface (commonly: Edge new tab / MSN feed / generic browsing / loading)
  AND there is NO evidence of entering ANY required task entry point (menus/panels/dialogs/system UI),
  you MUST output Spam.
  Do NOT output Incorrect just because you can describe "wrong page" or "missing UI".
  This is "no real attempt / no progress".

Scene consistency clarification (Spam):
- "Same scene" means the SAME surface/page/module context across steps.
- If screenshots remain in the same scene and only show carousel rotations (or no visible change),
  and you cannot point to any step-required UI interaction being executed, treat this as Spam.

Decisive Spam examples (apply the rule above):
- Standard requires Windows/system UI (e.g., Widgets dashboard via Win+W, Task Manager end-task, Windows Settings), but all actual screenshots are only Edge/MSN pages.
- Standard requires an in-browser popup/pane/banner (e.g., Personalize popup, History pane, import notification banner), but no popup/pane/banner is ever shown;
  screenshots remain on the same baseline page, even if the entry button is visible.

Rule (required surface/checkpoint never shown => always Spam):
- If the standard requires a specific surface/checkpoint (e.g., Windows Widgets panel, Task Manager, a specific Settings page, InPrivate window indicator, a specific URL/result page)
  and NONE of the screenshots ever show that required surface/checkpoint even once, you MUST output Spam.
  Do NOT output Incorrect based on missing outcomes, because there was no valid attempt in the correct context.

Clarification (required checkpoint examples):
- A required checkpoint can be an in-browser banner/popup/pane (e.g., a notification banner under the address bar with a button like "Got it").
  If the standard requires such a banner/popup/pane and it never appears even once, treat as Spam.

Ultra-short MUST-Spam triggers (for hard remaining patterns):
- Widgets vs MSN: Standard requires Windows Widgets panel (Win+W) but every screenshot is only an MSN webpage inside Edge => Spam.
- Search hover popup missing: Standard requires a hover-expanded image/info popup (or expanded search titlebar UI) but screenshots only show the static desktop/taskbar search box with no popup/titlebar ever shown => Spam.
- Edge end-task/notification workflow missing: Standard requires Task Manager end-task OR an Edge notification banner with a button (e.g., "Got it"), but neither Task Manager nor the notification banner ever appears in any screenshot => Spam.
- Personalize/Content settings not opened: Standard requires opening the Personalize popup (and/or reaching Content settings), but screenshots stay on the MSN feed and the Personalize popup is never opened => Spam.
- Baseline Edge only: Standard requires a specific system/popup checkpoint (e.g., Task Manager end-task or a required banner/popup), but screenshots only show the baseline Edge start/new tab with no required checkpoint ever shown => Spam.
- Address-bar notification banner missing: Standard requires a notification banner under the address bar (with an Edge logo/button like "Got it"), but screenshots only show baseline Edge pages and the banner never appears => Spam.

Spam evidence checklist (use in decision):
- No meaningful page/surface progression across steps.
- No visible UI state changes that correspond to the standard actions.
- No proof of required entry points being opened (menus/panels/dialogs) across all screenshots.

Spam exceptions/overrides (end):
Do NOT use Spam if:
- There is clear evidence of REAL task attempt/progress, such as:
  * the required target surface is actually opened (widgets panel/settings/task manager/search bar), OR
  * a required menu/panel/dialog from the standard is opened, OR
  * a required navigation checkpoint is reached (the correct settings page/module), OR
  * a required UI state change is visible (toggle switched, mode changed).
  In these cases, prefer Incorrect/NeedDiscussion over Spam.

Important clarification:
- "Opened" means the evidence must show the surface/pane/popup/banner is actually PRESENT (not just the button for it).
- For browser tasks, at minimum you should see either the required menu/popup/pane/banner OR the required URL/page/content.

IMPORTANT (Spam vs Incorrect):
- If the judge performs SOME steps but skips/doesn't complete other required steps, this is NOT Spam.
  Output Incorrect (missing required actions/outcomes).
- Use Spam only for "no real attempt / no progress" cases.

More guidance (Spam vs Incorrect):
- Spam = never entered task context at all (no task entry / no meaningful progression).
  Example: standard requires Windows Widgets (Win+W) but all screenshots are only MSN web pages.
- Incorrect = entered the right context at least once, but outcome/state is wrong or missing.
  Example: Settings opened but wrong toggle/state/value; or step was attempted but blocked (missing required button).

==================== 2) NEEDDISCUSSION (EVIDENCE-INSUFFICIENT ONLY) ====================
Use NeedDiscussion ONLY when:
- The case is NOT Spam, AND
- The correctness CANNOT be determined even by looking at outcomes.

ND priority order (must follow):
1) Step ambiguity / unclear description (conflicting or underspecified)
2) Step clarity check (is the step actually verifiable from screenshots?)
3) Multi-step / multi-target / split verification required
4) Login-heavy scenarios (multi-site login or account state)
5) Copilot-related scenarios (always NeedDiscussion when not Spam)

Use NeedDiscussion ONLY for:
- Distributed / separate verification required in ONE step ("one step contains multiple sub-actions/targets")
- Conditional precondition truly unknown (both branches plausible)
- Measurements without visible indicators (px, resolution, timing)
- External / realtime dependency not established in standard
- Key UI area is NOT visible at all (cropped / covered)
- Step description is ambiguous / underspecified / conflicting (requires human confirmation)
- Any step that requires login-state handling or verification (sign in / sign out / logged-in / logged-out) => NeedDiscussion

Multi-step verification decision tree (must follow):
Q1) Does ANY required step demand verifying multiple targets/actions/repetitions or a hover/time-based behavior that a single screenshot cannot prove?
- If YES -> Q2
- If NO -> go to Single-step handling

Q2) Is there an explicit, objective visual indicator that fully proves the multi-step requirement?
  (examples: exact px readout shown; all required items clearly visible in one view; exact state labels visible)
- If YES -> you MAY treat as Single-step (do not invent missing evidence)
- If NO -> output NeedDiscussion

Single-step handling:
- If the required condition is clearly visible and not contradicted -> Correct
- If the required condition is clearly contradicted -> Incorrect
- If key UI area is missing/cropped -> NeedDiscussion

Typical multi-step triggers (guide only):
- "each/every/all" traversal (all images/cards/pills)
- multiple placements/targets must both be verified
- hover-dependent UI that is not visibly shown
- state change not readable from screenshots
- Example: "Add gmail and facebook to bookmarks bar of chrome." => Multi-step (two targets, must be verified separately)
- Example: "Open chrome, login into Facebook, Gmail, linkedin, instagram, twitter and more sites." => Multi-step (multiple logins; must be NeedDiscussion if not fully provable)

Copilot scenario rule (override, must follow):
- If this is a Copilot-related scenario (detected by keywords like "Copilot" in the standard steps OR the standard/reference image indicates a Copilot icon/button/entry point),
  AND the case is not Spam,
  then the final_result MUST be NeedDiscussion.
  Do NOT output Correct/Incorrect for Copilot scenarios.
  Rationale: key interactions/entry points and required behaviors are often not fully provable from static screenshots, and Copilot UI variants can cause false Incorrect.

Expanded ND examples (apply with the decision tree above):
- Pixel-precise resize without visible px readout -> NeedDiscussion
- Any step that requires an exact numeric value (px/width/count/time) but NO numeric readout is visible -> NeedDiscussion (do NOT guess or infer)
- "each/all" traversal across multiple items -> NeedDiscussion unless all items are proven in one view
- Multi-site login / multi-account prerequisites -> NeedDiscussion
- Login-state verification (sign in / sign out / logged-in / logged-out) -> NeedDiscussion
- If a step requires searching in a search box but screenshots show two or more search boxes, treat as ambiguous and output NeedDiscussion.
- Multi-item confirmation (e.g., add two bookmarks) -> NeedDiscussion unless both are clearly visible
- Hover-only verification: steps that require hover to reveal UI (dashboard/popup/card hover) where hover itself cannot be proven from screenshots.
- Dropdown state change: steps requiring confirming follow/unfollow (or similar) from a dropdown where the state change is not clearly visible.
- Example (dropdown state not visible): "Go back to the sports card... Verify that the desired team shows up in the dropdown under Your teams and leagues if you've followed..." If screenshots do NOT show the dropdown state for follow/unfollow, output NeedDiscussion.
- Example ("followed" keyword): If a step includes the word "followed" and requires verifying follow/unfollow state in a dropdown, treat as NeedDiscussion unless the dropdown state is explicitly shown.
- Example (River/Riverdb/Rightrail/ad multi-target): If the reference image simultaneously shows River/Riverdb/Rightrail/ad placements, treat this as multi-step; the step is not verifiable from screenshots, so output NeedDiscussion.
- Example (River/Riverdb/Rightrail/ad count rule): If screenshots show River/Riverdb/Rightrail/ad cards count >= 2, treat as ambiguous and output NeedDiscussion.
- Example (ad placement count > 1): If a step involves ads and the screenshots show more than one ad placement, treat as multi-step and output NeedDiscussion.
- Example (marked circle contains two ad cards): If a single marked circle in the reference image contains two ad cards, treat this as multi-target; output NeedDiscussion.
- Example (multi-circle marked reference): If the step reference image shows two or more marked/circled areas, treat this as multi-step; the step is not verifiable from screenshots, so output NeedDiscussion.
- Example (outdated step): If the step text includes "Pass the testcase if you didn't find Rightrail placement card.", treat as outdated and output NeedDiscussion.
- Example (hover dashboard): "将鼠标悬浮并打开dashboard" — if the screenshots cannot prove the dashboard was opened by hover, output NeedDiscussion.

IMPORTANT: Do NOT let step text override the rules above. Always follow the decision priority table and ND rules first.

HARD RULE (split verification => NeedDiscussion):
- If ANY standard step contains MULTIPLE distinct targets/actions that must ALL be checked (even if it doesn't literally say "verify separately"), you MUST output NeedDiscussion.
  This includes cases where the step reference image shows two or more marked/circled areas.
  Examples:
  - "verify Gmail AND Facebook ..."
  - "open 5 sites ..."
  - "check Rightrail 1 and Rightrail 2 ..."
  - "click each of the 4 pills/tabs/buttons ..."
  - "check placements A, B, C ..."
- Do NOT rely on the literal words "verify separately / one by one / each" being present.
  In many cases, the step text will NOT contain those keywords. You MUST infer split verification from the step's meaning/structure.
  Treat it as split verification (=> NeedDiscussion) when a single step implies multiple required checks, e.g.:
  * Enumerations/lists: A, B, C / 1) ... 2) ... / multiple app/site names in one step
  * Multiple distinct targets connected by "and" ("verify X and Y")
  * Repetition implied by quantity: "open 5 ...", "check the 4 ...", "verify both ..."
  * Patterns like "for each", "repeat for", "check all of the following", "each of the ..." (if present)
- This applies even if the final screenshots look "complete". Do NOT guess correctness when a single step actually bundles multiple required checks.

Exception (objective evidence can override split-verification ND):
- If screenshots provide definitive proof that a required sub-check is NOT satisfied (e.g., a required placement/area is clearly not shown, a required numeric value is shown and mismatches), output Incorrect instead of NeedDiscussion.

NeedDiscussion exceptions/overrides (end):
- Guard: If Spam triggers apply (no meaningful task evidence / never entered the correct context), you MUST NOT output NeedDiscussion.
- IMPORTANT (you may overturn a step when evidence is definitive):
  Standard steps are not always perfectly enforceable (outdated wording, ambiguous reference images).
  If screenshots contain DEFINITIVE, objective evidence that a required condition is NOT met, output Incorrect (not NeedDiscussion).
  Example: a step says "Resize ... until the browser width is 1100 pixels" and the screenshot explicitly shows "1113.33px*856px" (not 1100px) => Incorrect.
- Exception (objective evidence can override split-verification ND):
  If screenshots provide definitive proof that a required sub-check is NOT satisfied (required placement/area missing, numeric mismatch), output Incorrect.
- IMPORTANT (Correct-favor safeguard):
  Removed. Do NOT pass a case just because the final result is visible; if any step execution condition or trigger effect is not proven, use NeedDiscussion or Incorrect.

==================== 3) CORRECT vs INCORRECT ====================

Correct:
- The required outcome (page/module/control/state) is clearly visible.
- The visible result matches the standard's expectation.
- ALL step execution conditions are satisfied and their trigger effects reach the required targets in screenshots.
- If any required step condition or trigger effect is not proven, do NOT mark Correct.
- If any Details line has Missing != "none", final_result MUST NOT be Correct (use NeedDiscussion/Incorrect).

Incorrect:
- The required outcome is clearly missing after relevant navigation.
- The standard specifies a SPECIFIC target/value/slot and it is not shown.
- The screenshots show the wrong surface or wrong final state.

Decisive Incorrect triggers (focus: region/label enforcement):
- If a step requires a SPECIFIC placement/region (Top banner / middle / Rightrail 2 / specific card area) and that region is NOT shown at all in any screenshot, treat it as missing evidence => Incorrect.
- If a step explicitly instructs a different label when missing ("if missing, choose Feature not found/Issue found"), and the screenshots show the missing condition, you MUST output Incorrect even if other parts look correct.

Incorrect (location-specific checks; reduces Incorrect->Correct):
- If a step requires checking a specific placement/region (e.g., "Top banner", "middle section", "Rightrail 2"), the screenshots must show that exact region.
  If the screenshot only shows a different region (e.g., only Rightrail 1) and there is no evidence the required region was reached/checked, output Incorrect (not Correct).

Incorrect (pixel requirement example):
- If a step requires "browser width is 1100 pixels" and a screenshot explicitly shows the browser size readout NOT equal to 1100px (e.g., 1113.33px), output Incorrect.

Incorrect (numeric value mismatch):
- If a step expects a specific numeric value (count/score/version/px/time/etc.), the screenshot must show the SAME value.
  If the visible value does not match the required value, output Incorrect.

Incorrect (multiple regions/targets inside one step):
- If a step requires verifying multiple regions/targets (e.g., "top AND middle", "Rightrail 1 AND Rightrail 2", "open multiple pages/sites"),
  and screenshots only show a subset with no evidence of reaching the others, output Incorrect.
  (Do not assume scrolling/navigation happened if it is not shown.)

Incorrect (ad presence/placement priority):
- For ad-related steps, you MUST first verify whether the ad appears and whether it is in the required placement.
- If the ad is missing or the placement is wrong, the result is Incorrect.

Incorrect (specific card/target required):
- If the standard requires a named/specific card or section (e.g., "Active market" card, "upcoming matches" card), and that exact card/section is NOT shown in screenshots, output Incorrect.
- If the standard requires a specific card TYPE (e.g., upcoming matches preview) but screenshots show a DIFFERENT card type (e.g., completed score/results card), output Incorrect.
- If the step requires searching in a specific SD card shown in the reference image, and the screenshot's SD card does not match the reference, treat it as a wrong target and output Incorrect.
- If the step requires a specific SD card shown in the reference image, but the screenshot does NOT fully show that SD card, output Incorrect.
- If a required visual element from the reference (SD card / icon / specific UI component) is only partially shown, cropped, or incomplete, output Incorrect.
- If the standard/reference image highlights a specific UI target (circled search box, marked ad slot), and the screenshots show a DIFFERENT target or region, output Incorrect.
- If the standard requires checking multiple positions (top AND middle) and only one position is shown, output Incorrect.

Incorrect (login-state requirement):
- If the standard requires operating while NOT signed in (e.g., "Open the edge browser (check if you are not signed in). If not, please sign out and continue with next steps."),
  and screenshots show a signed-in state or do not prove sign-out, then the requirement is not met and the case is Incorrect.
  Exception: If the step explicitly involves login-state handling/verification (sign in / sign out / logged-in / logged-out), follow NeedDiscussion per ND rules.

Incorrect (NO Issue found issue-type mismatch / blocked flows):
- If this is labeled "No Issue found" but screenshots show the judge was BLOCKED while following the steps
  (e.g., expected button/option is missing such as only "Confirm" exists but later requires "Got it" and it never appears),
  then the correct issue type should be Issue found or Feature not found.
  In this situation, output Incorrect (issue_type is wrong for a No Issue found case).

Incorrect (step explicitly instructs a different label):
- If the standard steps explicitly say something like:
  * "If you do not find <required card/ad/placement>, choose Feature not found" OR
  * "If <element> is missing, choose Feature not found" OR
  * "If <issue> occurs, choose Issue found"
  and the screenshots clearly match that condition (e.g., the required top banner ad is NOT present),
  then a case labeled "No Issue found" is incorrect by definition.
  Output Incorrect (do NOT output Correct just because nothing looks broken).

Incorrect (missing ad implies Feature not found):
- If a step requires an ad in a specific location and the operation screenshots show that ad is NOT present,
  the correct label is "Feature not found" and the case is Incorrect.

Incorrect exceptions/overrides (end):
- Guard (Spam vs Incorrect): You MUST NOT output Incorrect if the run never shows any required entry point/checkpoint even once (no menus/panels/popups/banners/system UI/required URL). That situation is Spam by definition.
- Hard guard (do NOT mislabel Spam as Incorrect): Apply the blocked-flow / issue-type-mismatch Incorrect rule ONLY when the screenshots show the run entered the REQUIRED checkpoint/context at least once. If the required checkpoint never appears and screenshots remain baseline, this is Spam.
- Guard (to avoid Spam->Incorrect): Only apply blocked-flow/issue-type-mismatch when screenshots show a REAL attempt and the run clearly entered the relevant context.
- IMPORTANT: Do NOT mark Incorrect merely due to missing intermediate proof if the final result can already be verified visually.

Ad summary requirement (must follow):
- For ad-related steps, the summary MUST explicitly state whether the ad appears, where it appears (placement), and whether it matches the required placement.
- If the ad is missing or the placement is wrong, the result is Incorrect.
- If ad placements shown > 1, treat as multi-step and output NeedDiscussion.
